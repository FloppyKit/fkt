## 1. Offline Ark / Bark: what FKT can reasonably sign

Bark (Second’s Ark) is **clArk** (covenant-less): trees of pre-signed / MuSig2-server-collaborative txs, not CTV. FKT is a **single BIP39 offline signer**. That filters hard.

### Offline-reasonable (user key only, after we build script-path)

| Flow | What happens | Offline FKT role | FKT build needed |
|------|----------------|------------------|------------------|
| **Board funding** | On-chain spend from bark’s BDK wallet → board address | **Yes** if PSBT is exported with BIP32 paths for the same mnemonic | None beyond v0.1 if board inputs are keypath P2TR/P2WPKH; need **PSBT export hook** (bark signs internally today) |
| **Exit claim (user leaf)** | After exit txs confirm + CSV/CLTV, claim via **script-path**: witness `[schnorr_sig, tapscript, control_block]` on leaves like `DelayedSign` / `TimelockSign` / `DelayedTimelockSign` where `pubkey` is the **user** | **Yes — primary offline value** | Taproot **script-path** parse + BIP341 leaf sighash + untweaked schnorr + final_scriptwitness |
| **Exit package CPFP / fee bump** | On-chain fee child from user UTXO | **Yes** if standard keypath PSBT with paths | v0.1 already if keypath |

### Not reasonable as pure offline single-sig (do not promise)

| Flow | Why |
|------|-----|
| **Round / refresh / pool** | MuSig2 + ASP nonces; multi-party; not one seed alone |
| **Cooperative offboard** | Forfeit + server cosign exchange (online protocol) |
| **Arkoor send** | Server cosign delivery; off-chain mailbox |
| **Lightning send/recv** | Server + HTLC; hashlock needs preimage orchestration |
| **VTXO tree node keyspends** | Aggregate user+server keys; bark wallet even refuses plain keyspend on policy leaves |
| **“Covenant” CTV boards** | Not what bark implements on Bitcoin today |

### Practical Ark roadmap for FKT (when faucet works)

1. **Phase A (build):** script-path single-sig leaves (see prior Ark plan).  
2. **Phase B (fixtures):** synthetic exit-claim PSBTs with known seed.  
3. **Phase C (live):** fund bark → export mnemonic → load same seed in FKT → capture claim PSBT (or reconstruct from `raw-vtxo`) → FKT sign → broadcast claim.  
4. **Board:** only if we add a “dump unsigned board funding PSBT” path (upstream bark change or BDK intercept); otherwise board stays “bark-internal.”

**Faucet note:** Second’s `tark1` UI currently fails bech32m parse; on-chain is `tb1p` (many faucets reject). **Do not block testing plan on Ark live funds.**

## 3. What is left to *build* (vs just *generate PSBTs*)

### A. No new crypto (v0.1 validation lane) — **highest ROI**

1. **Normalize sparrow-real layout** to skill structure (or document mapping):
   - `unsigned/`, `sparrow-signed/`, `fkt-signed/`, `manifest.json`, `README.md`
2. **`manifest.json` schema** per case:
   ```json
   {
     "id": "p2wpkh_1in_1out",
     "unsigned": "unsigned/p2wpkh_1in_1out.psbt",
     "expect": "sign",
     "script": "p2wpkh",
     "seed_env": "FKT_SPARROW_SEED",
     "notes": "BIP84 baseline"
   }
   ```
3. **`make test-sparrow-real`** harness:
   - For `expect=sign`: load seed from env, sign, write `fkt-signed/`, optional compare sighash/witness shape to sparrow-signed
   - For `expect=reject`: assert non-zero + known error substring
4. **Create missing High Sparrow PSBT:** `mixed_p2wpkh_p2tr_2in_2out` (you + Sparrow)
5. **Fill gaps:** nested P2SH-P2WPKH real unsigned; optional 2-of-2 for reject/partial
6. **Policy decision (document in manifest):**
   - Nested P2SH-P2WPKH: **sign if key matches** (code can) *or* **reject in UI** for v0.1 simplicity  
   - P2WSH: **partial_sig only** / “not fully supported” message

### B. Build next (0.2) — enables Ark exit offline

1. Remove hard abort on PSBT `0x18` merkle root  
2. Parse `TAP_LEAF_SCRIPT` (0x15) / control block / script-path flags  
3. BIP341 **script-path** sighash (ext_flag, tapleaf_hash, key_version)  
4. Sign untweaked key for single-sig leaves; finalize `[sig, script, control_block]`  
5. Synthetic `p2tr_script_path` + later Ark exit-claim fixtures  

### C. Build later (0.3 / Ark)

1. Proprietary key **passthrough** (don’t strip `bark` ClaimInput)  
2. Preview labels for exit/board  
3. Live capture scripts once faucet works  
4. **Not** MuSig rounds / forfeit protocol inside FKT  

### D. Explicitly not building for this plan

- Camera, PWA, full V2 cosign UX, CTV covenants, bark protocol state machine
